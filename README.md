# RaccoonHouse Studio

Десктопний застосунок для студії дубляжу аніме на Windows: керування тайтлами й
серіями, ізоляція вокалу нейромережею, робота з субтитрами (ASS/SRT), передача
проєкту в Reaper, і опційна розподілена обробка між ПК студії.

## Що і як працює — короткий огляд

Типовий шлях однієї серії через застосунок:

1. **Тайтли** — створіть тайтл (назву можна ввести вручну або знайти постер
   через Hikka — див. "Постери" нижче). Тайтл видаляється кнопкою в його ж
   плитці (разом з усіма серіями).
2. **Імпорт відео** — перетягніть відеофайл у вікно тайтлу (mp4, mkv, avi,
   mov, m2ts, ts — ffmpeg сам розбирає контейнер, окремої підтримки формату
   не треба). Номер сезону й серії можна виправити вручну перед імпортом
   (застосунок лише вгадує номер із назви файлу як стартову підказку). Далі:
   **ffmpeg** витягує лише лосслес-аудіодоріжку (FLAC) — див. "Імпорт відео"
   нижче. Якщо цей ПК слабкий, тут-таки можна попросити, щоб цей крок виконав
   потужніший ПК студії — див. "Розподілена обробка потужності".
3. **Ізоляція вокалу** — окрема кнопка в робочому вікні серії, обирається
   модель (MDX-Net / VR Arch / Demucs / MDX23C / BS-RoFormer) чи Ensemble
   Mode; так само може виконатись локально або на іншому ПК.
4. **Субтитри** — імпорт ASS/SRT прямо в робоче вікно серії; теги
   оформлення (Sign/Signs/OP/ED) розпізнаються окремо, перекриття рядків і
   помилки часових кодів підсвічуються.
5. **Маркери персонажів** — автоматичне виявлення пауз/реплік (VAD) на
   вокальному стемі, з прив'язкою до кодів персонажів для орієнтиру в Reaper.
   Замість того щоб щоразу вручну набирати код персонажа в назві маркера,
   кнопка-іконка поруч з полем назви відкриває список уже наявних персонажів
   тайтлу — код додається в назву одним кліком.
6. **Передача в Reaper** — експорт CSV-маркерів (і за потреби ReaScript) для
   імпорту тайм-лінії прямо в проєкт Reaper.
7. **Фінальний рендер** — після зведення дубляжу в Reaper, готовий мікс
   імпортується назад і мультиплексується (`ffmpeg`, відео копіюється
   без перекодування) проти оригінального відео.

Усі довгі операції (ffmpeg, нейромережа, детекція маркерів, віддалені
запити) виконуються у фоні з наживо видимим відсотком і статусом — у
плитці серії, в робочому вікні серії, і компактно в шапці вікна.

## Архітектура

- **Оболонка/UI**: Electron + React + TypeScript + Tailwind (`src/`, `electron/`)
- **Бекенд**: локальний FastAPI-сервер (`backend/`), запускається процесом Electron,
  зберігає дані в SQLite в системній user-data теці
- **Ізоляція вокалу**: [`audio-separator`](https://pypi.org/project/audio-separator/)
  (обгортка над моделями UVR — MDX-Net / VR Arch / Demucs / MDX23C / BS-RoFormer)
- **ffmpeg**: вбудований бінарник — лише витягування аудіо при імпорті й
  фінальний мультиплексинг; жодного перекодування/стиснення відео ніде в
  застосунку немає — перегляд завжди йде з оригінального файлу
- **Пакування**: electron-builder (NSIS) + PyInstaller (Python-бекенд збирається
  в окремий `.exe`, тож користувачу не потрібен встановлений Python)

Electron запускає бекенд як дочірній процес при старті й **примусово завершує
все дерево процесів** (`taskkill /T /F` на Windows) при закритті вікна чи виході
із застосунку — бекенд не лишається висіти у фоні після закриття програми.

## Розробка

```
npm install
python -m venv .venv
.venv\Scripts\pip install -r backend\requirements.txt
npm run dev:full     # бекенд + Electron одночасно, в dev-режимі
```

> **ffmpeg**: `resources/bin/ffmpeg.exe` і `resources/bin/ffprobe.exe` **не** зберігаються
> в git (кожен ~240 МБ, більше за ліміт GitHub 100 МБ/файл). Перед першою збіркою
> скопіюйте їх туди з будь-якого локального встановлення ffmpeg — на dev-машині
> `where ffmpeg` покаже поточний шлях.

## Збірка інсталятора

```
npm run build:exe
```

Це запускає PyInstaller (бекенд → `backend-dist/raccoonhouse-backend.exe`), збирає
рендерер (`dist/`) та Electron main/preload (`dist-electron/`), потім пакує все
через electron-builder у `release/` — розпакований застосунок лежить у
`release/win-unpacked/RaccoonHouse Studio.exe`, а якщо крок пакування пройшов
успішно, поруч з'являється однофайловий інсталятор
`release/RaccoonHouse Studio Setup <версія>.exe`.

> **Windows Developer Mode**: крок electron-builder з NSIS створює символічні
> посилання під час розпакування допоміжного архіву для підпису коду. Windows
> блокує це для не-адміністративних процесів, якщо не увімкнено Developer Mode
> (Параметри → Конфіденційність і безпека → Для розробників). Це одноразове
> системне налаштування, потрібне лише під час *збірки* інсталятора, не під час
> роботи застосунку.

## Де зберігаються дані

Усе, що застосунок записує, лежить у системній user-data теці (`RH_DATA_DIR`,
зазвичай Electron-івський `userData`):

- `raccoonhouse.db` — SQLite (тайтли, серії, персонажі, дублери, субтитри,
  маркери, профілі, налаштування)
- `episodes/<id>/` — аудіодоріжка серії, вокальний стем (перегляд відео йде
  напряму з оригінального файлу, тут нічого не зберігається)
- `models/` — завантажені ваги моделей audio-separator (кешуються після першого разу)
- `logs/app.log`, `logs/power_share.log` — див. нижче

## Імпорт відео: один швидкий ffmpeg-крок, без транскодування

При імпорті відео виконується **лише витягування лосслес-аудіодоріжки
(FLAC)** — прогрес-бар 0–100%, "Витягую аудіо… 40%". Жодного 480p-проксі
більше **не генерується**: перегляд у робочому вікні йде напряму з
оригінального файлу (через `/api/stream`), а не з транскодованої копії. Це
означає, що субтитри, маркери й акторів можна виставляти одразу, щойно цей
єдиний ffmpeg-крок завершився — не чекаючи на другий, повільніший
прохід перекодування, і паралельно з ізоляцією вокалу, яка на цьому етапі
ще навіть не почалась.

Ізоляція вокалу нейромережею **не запускається автоматично** — це окрема дія
за натисканням кнопки "Розпочати відокремлення вокалу" в робочому вікні серії.
Це дає змогу обрати модель (або залишити ту, що обрана в Налаштуваннях),
а також — якщо ввімкнено розподілену обробку потужності — попросити виконати
розділення на потужнішому ПК замість свого (кнопка "Попросити потужності").

Прогрес відокремлення так само видно наживо, а після завершення робоче вікно
й плитка серії **оновлюються самі** — не потрібно виходити з серії й заходити
знову, щоб побачити новий стем, субтитри чи інший результат фонової задачі
(включно з акторами: якщо фонова задача додала нових персонажів — наприклад,
імпорт ASS — список акторів у випадаючому списку теж оновлюється сам).

Якщо серію видаляють, поки для неї ще триває фонова задача (імпорт,
відокремлення тощо) — ця задача негайно скасовується і зникає з індикатора
в шапці вікна, а не продовжує показувати застарілий відсоток нескінченно.

> Якщо оновлюєтесь із версії, де проксі ще генерувався: старі
> `proxy_480p.mp4`-файли нікуди не зникають самі — кнопка "Очистити" в
> Налаштуваннях → "Кеш прев'ю (480p)" й далі прибирає їх з диска, просто нові
> серії такого кешу вже не створюють.

## Розподілена обробка потужності

Ізоляція вокалу важка для CPU/GPU. Якщо один ПК студії слабкий, а інший
потужний, розподілена обробка дозволяє слабкому ПК попросити потужніший
виконати відокремлення вокалу (чи ffmpeg-імпорт) замість себе — **повністю
через інтернет, без жодного прямого з'єднання між двома ПК** — з явною
згодою реальної людини на іншому боці.

**Вимкнено за замовчуванням.** Доки перемикач у Налаштуваннях → "Розподілена
обробка потужності" вимкнено, ця функція взагалі нічого не робить.

### Як це працює — усе через один Cloudflare Worker

Немає ні локальної мережі, ні проброса портів, ні VPN-мостів, ні ручного
введення чиєїсь IP-адреси — все, що потрібно двом ПК, це бути одночасно
підключеними (через `wss://`) до одного невеликого Cloudflare Worker'а (див.
`cloudflare-signaling/`, налаштовується в Налаштуваннях → "Пристрої онлайн";
типово вже вказано на спільний Worker студії):

1. **Хто зараз онлайн** — Worker веде живий список підключених ПК і
   розсилає його всім, щойно він змінюється. Вкладка "Пристрої онлайн" в
   Налаштуваннях показує саме цей список.
2. **Згода** ("можна відокремити вокал на твоєму ПК?") — маленьке
   JSON-повідомлення, яке Worker пересилає від одного конкретного ПК до
   іншого через той самий WebSocket.
3. **Сам файл** (аудіо чи оригінальне відео вгору, готовий результат назад)
   — не йде через WebSocket і не йде напряму між ПК: відправник заливає
   файл у прив'язаний до Worker'а R2-бакет (`PUT /transfer/:id`), отримувач
   забирає його звідти (`GET /transfer/:id`), після чого обидві сторони
   видаляють об'єкт. Це навмисно "store-and-forward", а не пряма
   ретрансляція — Cloudflare Worker'и мають ліміт часу/CPU на один запит,
   що зробило б пряму ретрансляцію багатогігабайтного відео через сам
   Worker поганою ідеєю.

Розгортання власного Worker'а (якщо не використовуєте вже готовий) описано в
`cloudflare-signaling/README.md`.

### Що означає "доступний"

ПК вважається дійсно придатним для завдання, лише якщо на *ньому*:
1. Розподілену обробку увімкнено в Налаштуваннях, **і**
2. Хтось обрав активний профіль (аватар у сайдбарі — див. нижче).

Друга умова існує тому, що запит на згоду, який нікому показати, марний: якщо
профіль не активний, вхідні запити одразу відхиляються як `"no_profile"`, без
жодного спливаючого вікна.

### Хід запиту

Є дві точки входу — обидві запускають той самий механізм узгодження, лише з
різним корисним навантаженням:

- **До ffmpeg, у діалозі імпорту відео** — якщо є хоч один доступний ПК,
  з'являється позначка "Попросити потужність для ffmpeg"; на переданий ПК іде
  **оригінальний відеофайл**, там витягується аудіо, назад повертається лише
  готова аудіодоріжка (перегляд і так іде з оригінального файлу, локального
  чи ще не завантаженого — проксі більше нема). Для ПК, що насилу тягне навіть
  ffmpeg, це знімає все локальне навантаження ще до початку роботи.
- **У панелі відокремлення вокалу** — кнопка "Запросити потужність"
  (з'являється, лише коли розподілена обробка увімкнена на *вашому* ПК)
  надсилає вже витягнуту аудіодоріжку на віддалений ПК і отримує назад
  готовий вокальний стем.

Обидва варіанти:

1. Бекенд дивиться на кожен ПК, знайдений через Worker. Якщо якийсь ПК уже
   погоджувався на запит для **цього самого тайтлу** раніше — він
   використовується одразу, без повторного питання (див. "Запам'ятована
   згода" нижче).
2. Інакше — усім доступним ПК одночасно (не по одному) надсилається
   запит на згоду через WebSocket-релей Worker'а — саме так реалізовано
   "всім, хто зараз доступний, приходить сповіщення", а не лише одному
   вибраному.
3. На екрані кожного такого ПК WebSocket показує спливаюче вікно внизу справа:
   хто просить, для чого саме (імпорт відео чи відокремлення вокалу), для
   якого тайтлу/серії, з кнопками **Так / Ні** і 60-секундним відліком. Якщо
   ніхто не натиснув за 60 секунд, цей ПК автоматично відмовляє (`"timeout"`)
   і просто не використовується — це не блокує і не повторюється.
4. **Перший ПК, що погодився, перемагає** — решта запитів більше не
   очікується. Відправник заливає файл у R2-бакет Worker'а і повідомляє
   про це погодженого пірa через той самий WebSocket-релей; той ПК
   забирає файл із R2, виконує роботу локально (той самий пайплайн
   ffmpeg/audio-separator, що й при звичайній локальній обробці), заливає
   результат назад у R2 і повідомляє про завершення; відправник забирає
   результат і зберігає його так само, ніби робота відбулася на власній
   машині. Прогрес на екрані запитувача показує кожен етап окремо: пошук
   ПК → запит на дозвіл → передача файлу → очікування результату.
5. Якщо ніхто не погодився, запит завершується **конкретною причиною по
   кожному ПК** (вимкнено в налаштуваннях / ніхто не залогінений / відхилив /
   тайм-аут / не вдалося з'єднатися через сервер сигналізації) — не просто
   "не вдалося", а видно, що саме сталося і на якому ПК, щоб не гадати
   навмання.

**Що саме передається**: невеликий JSON-запит на згоду через WebSocket і,
якщо її надано, залежно від типу запиту — або оригінальне відео (вгору) та
аудіо (вниз), або вже витягнута аудіодоріжка (вгору) та готовий вокальний
стем (вниз), обидва через R2-бакет Worker'а. Жодні дані SQLite чи інші файли
проєкту не передаються.

### Запам'ятована згода (на рівні тайтлу, не запиту)

Щойно ПК погодився для конкретного тайтлу, цей самий ПК використовується для
*всіх наступних* серій *того самого* тайтлу без повторного питання — це
навмисно, щоб не спливало вікно на кожну з 24 серій сезону. Щойно робота
переходить на **інший тайтл**, весь цикл запитування всіх повторюється
заново. Це зберігається по парі (ПК, тайтл) у таблиці `power_share_consents`
— немає загального "довіряти цьому ПК завжди".

### "Загальна потужність" — графік

Вкладка "Загальна потужність" у Налаштуваннях показує графік потужності
кожного знайденого ПК: висота стовпчика — обсяг відеопам'яті (VRAM, ГБ) його
відеокарти, під кожним стовпчиком підпис із ніком людини та назвою відеокарти,
число ГБ написане прямо над стовпчиком. Ваш ПК завжди червоний (колір
акценту студії), кожен інший ПК має свій фіксований колір, пунктирна синя
лінія — середнє значення по всіх присутніх. Кольори підібрані та перевірені
на розрізнюваність для дальтонізму.

VRAM визначається через реєстр Windows (`HardwareInformation.qwMemorySize`),
а не через WMI `AdapterRAM` — останнє, як з'ясувалося на тестовій машині,
завжди занижує показник для відеокарт із понад 4 ГБ пам'яті (32-бітне поле).

### Мережева доступність

Бекенд слухає лише на `127.0.0.1` — жоден інший ПК ніколи не підключається
до нього напряму. Усе, що виходить назовні, робить це як звичайний вихідний
HTTPS/WSS-клієнт до Cloudflare Worker'а (той самий шлях, яким і браузер ходить
на будь-який сайт), тож локальний API (тайтли, серії тощо) не досяжний ні з
локальної мережі, ні з інтернету в жоден спосіб. Windows не повинна питати
дозвіл у брандмауері для цього застосунку.

### Логи

Кожен запит згоди/відповідь/таймаут і кожна віддалена обробка логуються в:

- `logs/power_share.log` — лише активність розподіленої обробки (хто кого
  просив, яке рішення, коли) — чистий журнал для аудиту саме цієї функції
- `logs/app.log` — ті самі записи впереміш із рештою того, що логує бекенд

Обидва — файли, що ротуються (2 МБ × 3 резервні копії), простий текст, один
рядок на подію.

### "Позичаю потужність" — індикатор

Коли цей ПК виконує відокремлення вокалу **для чужого запиту** (тобто хтось
інший попросив потужність, і саме ваш ПК погодився), зверху екрана — на
будь-якій вкладці, не лише в Налаштуваннях — з'являється невелика панель
"Допомагаю з потужністю: <тайтл>, серія <N> для <нік>". Вона зникає сама, щойно
обробка завершується. Це працює через те саме WebSocket-повідомлення
(`power_share_lending`), яке бекенд надсилає на початку й у кінці віддаленого
завдання.

## Автоматичне оновлення

Застосунок перевіряє нові версії через **GitHub Releases**
(`warpVIT1/raccoonhouse-studio`) за допомогою `electron-updater` — повністю
автоматично: перевірка йде одразу при старті й потім кожні 4 години, поки
застосунок відкритий, без потреби комусь натискати "Перевірити зараз"
(кнопка в Налаштуваннях лишається — для примусової негайної перевірки).
Якщо знайдено новішу версію, вона одразу завантажується у фоні.

Встановлення — **повністю тихе, без вікна інсталятора**: `quitAndInstall`
викликається з прапорцем silent-режиму NSIS, тож або людина сама натискає
"Перезапустити й встановити" (застосунок одразу закриється, тихо
встановиться і сам перевідкриється), або — якщо ніхто цього не зробив —
оновлення тихо встановлюється саме собою наступного разу, коли застосунок
закривають природним чином (`autoInstallOnAppQuit`). Жодного разу не
з'являється майстер встановлення з кнопками "Далі" — саме це й вимагалося:
"програма сама скачувала і встановлювала без відкриття інсталятора".
Оскільки в electron-builder.yml стоїть `perMachine: false` (встановлення в
теку користувача), тихе встановлення не вимагає прав адміністратора — жодного
запиту UAC.

Автооновлення працює лише у зібраному (упакованому) застосунку — у режимі
розробки (`npm run dev`) кнопка нічого не робить.

Щоб опублікувати нову версію: `npm run publish` (потрібен токен
`GH_TOKEN` з правом `repo` в змінних середовища) — це збирає бекенд,
рендерер і сам інсталятор, після чого завантажує їх у новий GitHub Release.

## Профілі користувачів

Аватар унизу сайдбару — це справжня локальна ідентичність цього ПК, а не
захардкоджене ім'я — натисніть, щоб створити/перемкнути/видалити локальний
профіль (ім'я, роль, колір). Саме це означає "залогінений" для розподіленої
обробки (див. вище). Зберігається в SQLite, окремо від списку `dubbers` (який
відстежує акторів озвучення, призначених на персонажів, а не хто керує
застосунком).

## Гарячі клавіші (робоче вікно серії)

Працюють, коли фокус не в текстовому полі (щоб не заважати звичайному
редагуванню):

- **Пробіл** — відтворити/пауза відео
- **← / →** — перемотка на 2 секунди назад/вперед
- **Shift + ← / Shift + →** — перемотка на 10 секунд назад/вперед

## Постери

Постер тайтлу можна знайти за назвою через публічний каталог
[Hikka](https://hikka.io) при додаванні/редагуванні тайтлу. Зображення **не
завантажуються на диск** — застосунок зберігає посилання на CDN Hikka напряму
й показує його наживо, тож постер потребує інтернету для показу (усе інше
працює повністю офлайн).
